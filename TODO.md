  ### Ограничения
  * Максимальный размер файла - 50MB (отклонять на клиенте до отправки)
  * Допустимые MIME типы:
    - Изображения: image/png, image/jpeg, image/webp, image/gif
    - Документы:   application/pdf
    - Данные:      application/xml, text/xml, application/json
    - Архивы:      application/zip, application/x-zip-compressed
  * Размер чанка - 48KB бинарных данных = ~64KB base64
  * Максимум 3 одновременных исходящих передачи на клиента (очередь остальных)

  ### Флоу загрузки (отправитель)
  1. Пользователь выбирает файл - проверить размер и MIME тип, отклонить если не подходит
  2. Вычислить SHA-256 через SubtleCrypto (Web Crypto API, доступен в WASM через web-sys)
  3. Проверить наличие в IndexedDB по хешу - если уже есть, только отправить FileAnnounce
  4. Сохранить blob в IndexedDB: { hash, mime_type, file_name, size, blob }
  5. Для файлов используемых в сценах - добавить FileRef в RoomState, вызвать commit_changes()
  6. Для файлов из чата - FileRef хранится только в ChatMessagePayload, не в RoomState
  7. Отправить FileAnnounce в WS

  ### Флоу получения (получатель)
  1. Получить FileAnnounce - проверить IndexedDB по хешу
  2. Если хеш уже есть - игнорировать
  3. Если нет - отправить FileRequest
  4. Дождаться начала получения чанков (таймаут 10 сек - если нет ответа, повторить запрос)
  5. Собирать чанки в Map<chunk_index, data> в памяти
  6. После получения всех чанков - собрать blob, вычислить SHA-256, сверить с hash
  7. Если хеш совпал - сохранить в IndexedDB, обновить UI
  8. Если не совпал - выбросить, отправить повторный FileRequest

  ### Выбор отвечающего (избегаем дублирования)
  * На FileRequest отвечает только один участник комнаты
  * Каждый клиент при получении FileRequest вычисляет: hash(request.requester + file.hash) mod N
    где N - количество участников комнаты, отсортированных по username
  * Если мой индекс совпадает с результатом - я отвечаю, остальные молчат
  * Детерминированный выбор без координации - все придут к одному ответу независимо

  ### Приоритетная очередь исходящих сообщений
  * Разделить WS отправку на три приоритета:
    - High   - Ping, ChatMessage, VotingCast, SyncSnapshot - отправляется немедленно
    - Normal - Presence, VotingStart, SyncVersionAnnounce - небольшая задержка при перегрузке
    - Low    - FileChunk - пауза 10ms между чанками, вытесняется High и Normal
  * Реализовать как три отдельных mpsc канала с одним циклом отправки:
    loop { drain High -> drain Normal -> send one Low chunk -> sleep(10ms) }
  * Это даст ~4.8MB/сек на передачу файлов не блокируя остальной трафик

  ### Обработка обрывов
  * Если WS соединение закрылось в середине передачи - незавершённые чанки в памяти выбросить
  * При переподключении - клиент снова видит FileRef в RoomState или ChatMessage, отправляет новый FileRequest
  * Передача возобновляется с нуля (без resume - упрощает логику)

  ### Вход в комнату с существующими файлами
